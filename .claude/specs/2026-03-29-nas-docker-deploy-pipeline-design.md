# NAS Docker Deploy Pipeline Design

**Date:** 2026-03-29
**Status:** Approved

## Problem

Deploying Docker service config changes to a home NAS requires a manual workflow: edit locally, push to GitHub, SSH
into the NAS, pull changes, restart containers. This is tedious for frequent config updates (Glance dashboards,
Home Assistant automations, etc.) and error-prone.

## Goals

- **Edit, push, deploy**: pushing to `main` automatically deploys changed configs to the NAS
- **Per-service granularity**: only affected containers restart when their config changes
- **Low maintenance**: minimal moving parts, no new software on the NAS beyond what exists
- **Ansible for bootstrap only**: Ansible handles initial NAS setup and secrets, not day-to-day deploys

## Constraints

- NAS is reachable via Tailscale (not publicly exposed)
- Secrets come from Ansible Vault and are rendered during bootstrap (never stored in git)
- Home Assistant config will be consolidated from its separate repo into this one

## Design

### 1. Repository Structure

Restructure from a single `roles/docker/files/docker-compose.yml` to per-service directories:

```
services/
├── compose.base.yml              # Shared network definitions, extension fields
├── pihole/
│   └── compose.yml               # PiHole service definition
├── plex/
│   └── compose.yml
├── transmission/
│   └── compose.yml
├── sonarr/
│   └── compose.yml
├── radarr/
│   └── compose.yml
├── prowlarr/
│   └── compose.yml
├── flaresolverr/
│   └── compose.yml
├── autoplex/
│   └── compose.yml
├── homeassistant/
│   ├── compose.yml
│   ├── config/                   # HA config dir (mounted as /config in container)
│   │   ├── configuration.yaml    # Consolidated from separate repo
│   │   ├── automations.yaml
│   │   ├── secrets.yaml.j2       # Template, rendered during bootstrap
│   │   └── ...                   # Other HA config files
│   └── shared/                   # Shared volume for inter-service communication
└── glance/
    ├── compose.yml
    ├── config/
    │   ├── glance.yml
    │   └── home.yml
    └── assets/
        └── user.css
```

**Key decisions:**
- Each service directory contains its compose fragment and all config files it needs
- Home Assistant is consolidated from `connormason/homeassistant` into `services/homeassistant/`
- Config files live next to their service so GitHub Actions path triggers naturally identify what changed
- The existing `roles/glance/` role is removed (Glance config moves to `services/glance/`)

**Removed services:**
- **Caddy**: Currently commented out in docker-compose.yml and not actively used. Not migrated to `services/`. The
Ansible caddy-dnsimple image build task is also removed. If Caddy is needed in the future, add a `services/caddy/`
directory at that time.
- **PyAutoplex**: Replaced by the Go-based Autoplex. Not migrated. The Ansible pyautoplex image build task is removed.
- The `.env` template will be updated to remove `DNSIMPLE_OAUTH_TOKEN` (only needed by Caddy). If Caddy is re-enabled
later, the token can be re-added.
- The `caddy_data` named volume is removed from compose definitions.

**Runtime data stays on the NAS, not in git:**
- PiHole's `etc-pihole/` directory is mutable runtime state (databases, DNS lists) generated by the running container.
It is NOT stored in the repo. The PiHole compose file mounts a host path (`~/docker/pihole-data/etc-pihole:/etc/pihole`)
that persists on the NAS outside of the rsynced service directory.
- Similarly, service config directories under `/storage/media/config/` (plex, sonarr, radarr, etc.) are runtime state
and remain on the NAS, referenced by absolute path in compose files.

### 2. Deploy Script (`scripts/deploy.sh`)

A bash script placed on the NAS during Ansible bootstrap. Handles the actual deployment work:

1. Receives service names as arguments (e.g., `deploy.sh glance homeassistant`)
2. For each service:
   - The service directory has already been rsynced to `~/docker/<service>/` by CI
   - Runs `docker compose --project-directory ~/docker -f ~/docker/compose.base.yml -f ~/docker/<service>/compose.yml up -d <service>`
3. If called with no arguments or with `all`, deploys every service
4. Handles `compose.base.yml` which defines shared networks/extension fields

**Service discovery for "all" mode:** The script discovers services by finding all subdirectories of `~/docker/` that
contain a `compose.yml` file. This naturally excludes non-service items (`.env`, `compose.base.yml`, and the `.build/`
directory used for custom image source repos).

**Working directory:** All `docker compose` commands use `--project-directory ~/docker` to set the project context.
This ensures:
- `env_file: .env` in compose files resolves to `~/docker/.env`
- Relative volume paths like `./glance/config:/app/config` resolve relative to `~/docker/`

The script does not handle change detection - that is GitHub Actions' responsibility.

### 3. Compose File Strategy

Each `compose.yml` is fully self-contained, defining the service, its volumes, ports, and network references.
The `compose.base.yml` at the services root defines shared resources.

On the NAS, compose files are merged at runtime:

```bash
docker compose --project-directory ~/docker \
               -f ~/docker/compose.base.yml \
               -f ~/docker/glance/compose.yml \
               up -d glance
```

**Why per-service files:**
- Any service change targets exactly that service's file
- Each file is small and focused
- `docker compose` natively supports multiple `-f` flags
- Service dependencies declared via `depends_on` resolve across merged files

If `compose.base.yml` changes, a full redeploy of all services is triggered.

**Volume paths:** All volume mounts in per-service compose files use absolute paths for data stored on the NAS
(e.g., `/storage/media/config/sonarr:/config`). Relative paths are used only when referencing files within the
service's own directory (e.g., `./glance/config:/app/config`), which resolve relative to the project directory
(`~/docker/`).

**Host networking:** Plex and Home Assistant use `network_mode: host` (Plex for media streaming compatibility,
Home Assistant for mDNS/UPnP service discovery). These services cannot join Docker networks and do not participate in
the shared network defined in `compose.base.yml`. Other services communicate with them via the host IP. This is the
existing behavior carried forward.

**Home Assistant volume layout:** HA config files live in `services/homeassistant/config/` in the repo, rsynced to
`~/docker/homeassistant/config/` on the NAS. The compose file mounts `./homeassistant/config:/config`
(resolved via `--project-directory` to `~/docker/homeassistant/config/`). This keeps `compose.yml` out of the
container's `/config` mount, unlike the current setup where the entire HA directory is mounted. The `shared/`
directory (`./homeassistant/shared:/shared`) provides inter-service communication, same as the current setup.

**Docker project name:** All `docker compose` commands use `--project-directory ~/docker`, which produces the project
name `docker`. To ensure consistency across individual service deploys, `compose.base.yml` sets `name: nas-stack`
explicitly. This prevents Docker from treating separate deploy invocations as different projects (which would create
duplicate networks and orphaned containers).

**Glance env_file:** Glance's `env_file: .env` references `~/docker/.env` (via project directory). This file currently
contains `FTLCONF_webserver_api_password` for PiHole. If Glance needs its own environment variables (API keys for
widgets, etc.), they should be added to the `.env` vault template. During migration, audit the existing `.env` on the
NAS to confirm what variables are currently set.

### 4. GitHub Actions Workflow (`.github/workflows/deploy.yml`)

Triggered on push to `main`. Two jobs:

**Job 1: `detect-changes`**
- Uses `git diff` between `github.event.before` and `github.sha` to identify changed files under `services/`. This
correctly handles multi-commit pushes (not just HEAD vs HEAD~1).
- **Fallback for null SHA:** On force pushes or new branches, `github.event.before` is the null SHA (`000...`). In
this case, the workflow treats it as a full deploy (all services). This is conservative but safe, and force pushes to a
personal dotfiles repo are uncommon.
- Outputs a JSON list of affected service names (e.g., `["glance", "homeassistant"]`)
- If `services/compose.base.yml` changed, outputs all service names
- If no services changed (e.g., only Ansible roles or docs edited), the deploy job is skipped

**Job 2: `deploy`**
- Runs sequentially (`max-parallel: 1`) over the service list from job 1. Sequential execution avoids race conditions
from concurrent `docker compose up` calls on the same host and ensures `compose.base.yml` is consistent across deploys.
- Steps:
  1. Checkout repo
  2. Set up Tailscale using `tailscale/github-action`
  3. Rsync `services/compose.base.yml` to `~/docker/compose.base.yml` on the NAS
  4. Rsync the changed service directory (`services/<name>/`) to `~/docker/<name>/` on the NAS
  5. SSH in and run `deploy.sh <service>`

**Rsync flags:** `rsync -avz --exclude='*.j2'` — the `--exclude='*.j2'` flag prevents Jinja2 templates (like
`secrets.yaml.j2`) from being synced to the NAS. Notably, `--delete` is NOT used. This prevents rsync from removing
files on the NAS that were placed there by Ansible bootstrap (rendered secrets, certificates) but don't exist in the
git repo. New files from git are added, changed files are updated, but NAS-only files (like `secrets.yaml` rendered
from `secrets.yaml.j2`) are preserved.

**Manual trigger:** The workflow supports `workflow_dispatch` with an optional service name input (defaults to `all`).
Useful for re-syncing after manual changes on the NAS.

### 5. GitHub Secrets

| Secret | Purpose |
|--------|---------|
| `TAILSCALE_OAUTH_CLIENT_ID` | Tailscale OAuth app client ID |
| `TAILSCALE_OAUTH_CLIENT_SECRET` | Tailscale OAuth app secret |
| `NAS_SSH_PRIVATE_KEY` | SSH private key for NAS access |
| `NAS_TAILSCALE_HOST` | NAS hostname/IP on the Tailnet |
| `NAS_USER` | SSH username on the NAS |

### 6. Ansible's Reduced Role

**Ansible still handles (bootstrap only):**
- Installing Docker engine and compose plugin
- Creating the `~/docker/` directory structure on the NAS
- Creating runtime data directories (e.g., `~/docker/pihole-data/etc-pihole/`)
- Building the custom autoplex image from its git repo (cloned to `~/docker/.build/autoplex` to avoid conflict with
the `~/docker/autoplex/` service directory)
- Rendering secret templates (`.env` from vault, `ha_secrets.yaml` from vault)
- Copying Lutron Caseta certificates from inventory
- Placing `deploy.sh` on the NAS
- Running an initial full deploy of all services

**Ansible stops handling:**
- Copying `docker-compose.yml` on every run
- Syncing Glance config files
- Building caddy-dnsimple and pyautoplex images (removed)
- Being the mechanism for day-to-day config changes

**Changes to existing roles:**
- `roles/docker/tasks/main.yml`: trimmed to remove config copy tasks and caddy/pyautoplex builds, replaced with
initial full deploy. **Important:** The `.env` rendering task is currently nested inside the "Setup caddy" block — it
must be extracted into its own top-level task before the caddy block is deleted.
- `roles/docker/defaults/main.yml`: remove caddy-dnsimple and pyautoplex repo variables; update `autoplex_repo_dest`
to `{{ docker_dir }}/.build/autoplex`
- `roles/docker/files/docker-compose.yml`: removed (replaced by per-service compose files in `services/`)
- `roles/docker/templates/env.j2`: updated to remove `DNSIMPLE_OAUTH_TOKEN`
- `roles/glance/`: removed entirely (config moves to `services/glance/`)

### 7. YAML Linting

Home Assistant uses non-standard YAML tags (`!include`, `!secret`) that standard YAML linters reject. The
`.pre-commit-config.yaml` yamllint configuration must be updated to exclude `services/homeassistant/config/` from YAML
linting, or add a `.yamllint.yaml` override within that directory that permits custom tags. Docker Compose YAML and
Glance YAML use standard syntax and need no special handling.

### 8. Secrets Handling

Secrets remain in Ansible Vault, rendered once during bootstrap:
- `.env` file (PiHole password) rendered to `~/docker/.env`
- `ha_secrets.yaml` rendered to `~/docker/homeassistant/config/secrets.yaml`
- Lutron Caseta certificates copied to `~/docker/homeassistant/config/`

The CI pipeline never touches secrets - it only syncs config files and compose definitions. The rsync
`--exclude='*.j2'` flag prevents templates from landing on the NAS, and the absence of `--delete` preserves rendered
secrets. To rotate a secret, re-run the Ansible bootstrap with the relevant tag.

### 9. Failure Handling

- Failed deploys show as red in GitHub Actions with full logs
- No automatic rollback - the previous container keeps running if `docker compose up` fails (Docker's default behavior)
- Failed workflows can be re-run from the GitHub UI, or fixed with a new push

## Day-to-Day Workflow

```bash
# Edit a config file
vim services/glance/config/home.yml

# Push to main
git add -A && git commit -m "update glance dashboard" && git push

# GitHub Actions automatically:
#   1. Detects services/glance/* changed
#   2. Connects to NAS via Tailscale
#   3. Rsyncs compose.base.yml and glance/ to NAS
#   4. Runs deploy.sh glance (docker compose up -d glance)
#   Done. ~30-60 seconds.
```

## Migration Path

1. Create `services/` directory structure with per-service compose files (absolute paths for NAS data, relative paths
for service-local config via `--project-directory`)
2. Create `services/compose.base.yml` with shared network and `name: nas-stack`
3. Consolidate Home Assistant config into `services/homeassistant/config/` from separate repo; include `shared/`
directory
4. Move Glance config from `roles/glance/` to `services/glance/`
5. Write `scripts/deploy.sh` with service discovery, `--project-directory` handling, and `COMPOSE_PROJECT_NAME`
consistency
6. Create `.github/workflows/deploy.yml` with change detection (null SHA fallback), sequential matrix, and rsync
(no `--delete`, exclude `.j2`)
7. Update `roles/docker/tasks/main.yml`: extract `.env` rendering from caddy block, remove caddy/pyautoplex builds,
update autoplex build path to `.build/`, remove config copy tasks, add deploy.sh placement and initial full deploy
8. Update `roles/docker/defaults/main.yml`: remove caddy/pyautoplex vars, update autoplex dest to `.build/autoplex`
9. Update `roles/docker/templates/env.j2` to remove `DNSIMPLE_OAUTH_TOKEN`; audit existing NAS `.env` for Glance
variables
10. Update `.pre-commit-config.yaml` yamllint to exclude `services/homeassistant/config/` (non-standard YAML tags)
11. Remove `roles/glance/` and `roles/docker/files/docker-compose.yml`
12. Set up GitHub secrets (Tailscale OAuth, SSH key, NAS host)
13. Test end-to-end: push a config change, verify automatic deploy
